# Human Label And Kappa Artifacts

**Phase:** P27 Human Label and Kappa Artifact Protocol Implementation

**Status:** EV3 artifact preparation only. P27 does not perform empirical
validation, run live APIs, fabricate labels, fabricate kappa, or claim
`measurement_validated`.

## 1. Purpose

P27 prepares the deterministic human-label and kappa artifact layer required
before any future measurement-validation review. It defines label schemas,
annotation templates, completeness checks, Cohen's kappa calculation, threshold
mapping, and deterministic kappa reports.

P27 is an input layer for future claim-gate evidence. It is not a replacement for
the existing conservative claim gate.

## 2. Label Schema

Required label fields:

- `run_id`
- `case_id`
- `condition`
- `annotator_id`
- `label_dimension`
- `label`
- `rationale`

Required label dimensions:

- `answer_correctness`
- `answer_completeness`
- `answer_groundedness`
- `context_sufficiency`
- `missing_critical_context`
- `irrelevant_context`
- `misleading_context`
- `conflict_or_stale_context`

Allowed label values:

| Label | Meaning |
| --- | --- |
| `0` | fail |
| `1` | partial |
| `2` | pass |

Templates generated by P27 are empty annotation templates only. They deliberately
leave `label` and `rationale` blank and must not be interpreted as completed
human labels.

## 3. Annotator Requirement

At least two annotators are required for kappa. If fewer than two annotators are
present, the kappa report records:

- `kappa_present: false`
- `measurement_validated_allowed: false`
- `two_annotators_required`
- `missing_kappa`

Additional annotators are handled by deterministic pairwise average kappa across
all annotator pairs. This avoids arbitrary annotator selection while preserving
stable output ordering.

## 4. Completeness Checks

The completeness checker reports:

- required and observed case counts;
- required and observed annotator counts;
- required and observed dimensions;
- missing cases;
- missing annotators;
- missing dimensions;
- duplicate label entries;
- invalid label values;
- `labels_complete`.

Rules:

- all required dimensions must be labeled;
- all validation-scoped cases and conditions must be labeled;
- at least two annotators are required;
- invalid label values fail closed;
- duplicate label entries fail closed;
- missing labels deny `measurement_validated`.

## 5. Kappa Computation

P27 computes Cohen's kappa without external dependencies.

For each label dimension:

1. Align items by `(case_id, condition)` for each annotator pair.
2. Compute observed agreement.
3. Compute expected agreement from marginal label distributions over `0`, `1`,
   and `2`.
4. Compute kappa as `(observed - expected) / (1 - expected)`.
5. Average pairwise kappas deterministically if more than two annotators are
   present.

The report includes:

- `macro_average_kappa`;
- `per_dimension_kappa`;
- `item_count_per_dimension`;
- `annotator_ids`;
- `pairwise_kappa`;
- `kappa_status`.

## 6. Threshold Mapping

| Kappa result | Status | Claim effect |
| --- | --- | --- |
| missing | `kappa_missing` | `measurement_validated_allowed: false` |
| `< 0.40` | `pilot_only` | Pilot-only or failed agreement evidence. |
| `0.40 <= kappa < 0.60` | `weak_evidence_not_measurement_validated` | Weak evidence; not validation. |
| `0.60 <= kappa < 0.75` | `limited_measurement_review_candidate` | Review candidate only if other gates later pass. |
| `>= 0.75` | `stronger_measurement_review_candidate` | Stronger review candidate only; not validation by itself. |

High kappa alone is not enough for `measurement_validated`.

## 7. Required Remaining Evidence

Even with complete labels and high kappa, validation still requires:

- complete artifacts;
- controlled live run evidence;
- contamination pass;
- fresh metric bridge;
- claim gate allow;
- P04 operator review;
- P09 operator review if runtime-integration claims are needed.

Missing contamination audit or missing/freshness-failed metric bridge keeps the
claim level below measurement validation.

## 8. Outputs

P27 can write:

- `human_label_completeness_report.json`
- `kappa_report.json`
- `kappa_report.md`

The compact summary fields intended for future P28 consumption include:

- `human_labels_present`
- `kappa_present`
- `labels_complete`
- `macro_average_kappa`
- `kappa_status`
- `measurement_validated_allowed`

`measurement_validated_allowed` remains `false` in P27 unless future external
evidence and the existing claim gate allow otherwise.

## 9. Claim Boundary

- P27 prepares EV3 human-label and kappa artifacts.
- P27 does not perform empirical validation.
- P27 does not fabricate labels.
- P27 does not fabricate kappa.
- Missing labels deny `measurement_validated`.
- Missing kappa denies `measurement_validated`.
- Low kappa maps to pilot-only or weak evidence.
- High kappa alone is not scientific validation.
- Contamination audit and fresh metric bridge remain required.
- P04 remains deferred/operator-required.
- P09 remains `BLOCKED_OPERATOR_REQUIRED`.
